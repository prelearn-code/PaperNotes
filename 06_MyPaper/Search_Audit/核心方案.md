# KCVB-Audit 核心方案

## 1. 方案定位

KCVB-Audit 面向加密去中心化存储中的关键词搜索与多副本完整性审计。公开验证者在一个冻结查询快照中验证：

1. 返回文件集合等于当前关键词的完整有效结果集合；
2. 每个结果文件的全部登记副本均进入挑战；
3. 每个响应绑定 `fid`、`copy_id`、块位置和当前块版本；
4. 当前关键词状态认证与多副本数据持有性进入同一个证书无关 pairing 等式；
5. 文件、索引、provider 和挑战状态在查询完成前不可替换。

核心构造只保留两个认证域：

- 数据域标签认证 copy-specific 编码块；
- 关键词状态标签认证当前 `ResultSetRoot_w` 和搜索 head。

完整结果集合由 sparse Merkle set 绑定，不为每条关键词关系生成独立代数标签。代数核心累加器 `(Sigma,R,z)` 为常数大小，但完整证明仍包含结果集合、副本分配、版本证明和 provider 局部响应，因此总通信和总验证时间不是常数。

## 2. 系统模型与声明边界

| 实体                      | 作用                        | 信任假设               |
| ----------------------- | ------------------------- | ------------------ |
| KGC                     | 生成公共参数和用户部分私钥             | 半可信，知道主密钥但不知道用户秘密值 |
| DO                      | 加密文件、生成副本、索引和标签           | 对自身数据和索引语义可信       |
| Search Provider, SP     | 保存前向私有索引并返回结果             | 不可信，可遗漏、替换或重放结果    |
| Storage Provider, `P_q` | 保存编码副本并生成局部响应             | 不可信，可删除、篡改、替换或合谋   |
| Blockchain/SC           | 保存 active roots、冻结查询并公开验证 | 共识和合约执行正确          |


SP 和全部 provider 可以合谋并共享历史 token、标签、挑战和证明。第一版限定为单 owner、单关键词、owner 发起搜索，并审计结果文件的全部 active 副本。

`owner_id` 是 certificateless 身份，不等同于交易发送地址。协议允许 relayer 提交索引更新；合约以 `kappa_w` 而不是 relayer 地址认证关键词状态所属 owner，避免把区块链账户签名当作重复的 owner 认证机制。

安全目标包括：前向私有搜索、完整结果集合、数据持有可提取性、副本独立性、块版本新鲜性、Type-I/Type-II 证书无关安全、审计响应标量隐藏、batch soundness 和 provider 归责。

允许泄露搜索模式、访问模式、结果数量、文件标识、副本数量、provider 分配和链上时序。方案不证明副本位于指定物理设备，也不阻止网络层转包。

## 3. 公共参数、密钥与接口

令 `G1,G2,GT` 为素数阶 `p` 的非对称双线性群：

```text
e: G1 x G2 -> GT
g1 in G1, g2 in G2, u in G1
```

定义域分离哈希：

```text
H_id:   {0,1}* -> G1
H_data: {0,1}* -> G1
H_kw:   {0,1}* -> G1
H_p:    {0,1}* -> Z_p^*
H:      {0,1}* -> {0,1}^lambda
```

KGC 选择 `s <- Z_p^*`：

```text
msk = s
mpk = g2^s
Q_ID = H_id("id" || ID)
D_ID = Q_ID^s
```

DO 选择 `x <- Z_p^*`：

```text
sk_DO = (D_ID,x)
pk_DO = g2^x
```

每个 provider 注册响应签名公钥 `pk_q`。已有文件固定使用创建时登记的 `pk_DO`。

```text
PrepareOutsource(F,KW,N,ProviderSet) -> (update_id,candidate_state)
AckStore(update_id,candidate_state,provider_id) -> StoreAck_q
ActivateOutsource(update_id,{StoreAck_q}) -> receipt

TokenGen(w,st_w) -> token_w
Search(token_w,EDB) -> (Res,Pi_idx)
FreezeQuery(Res,Pi_idx,c) -> (AuditRoot,beacon_round)
DeriveChallenge(AuditRoot,beacon_round) -> (seed_chal,chal)
CommitProof(chal,local_state) -> C_q
RevealProof(chal,C_all,local_state) -> response_q
Verify(Pi,query_state) -> {0,1}

PrepareBlockUpdate(fid,U_f,delta) -> (update_id,candidate_FileRoot)
PrepareRelationUpdate(op,w,RelCtx) -> (update_id,candidate_IndexRoot,Pi_idx_update)
PrepareReassign(fid,copy_id,new_provider) -> (update_id,candidate_FileRoot)
AckUpdate(update_id,candidate_root,provider_id) -> UpdateAck_q
ActivateUpdate(update_id,required_acks) -> active_root
AbortUpdate(update_id) -> unchanged_active_root
```

## 4. 文件、副本与认证状态

### 4.1 基础密文与独立副本

```text
fid = H("fid" || owner_id || file_nonce)
C_f = {c_{f,1},...,c_{f,n}}

copy_id_{f,j} = H("copy" || fid || j || copy_nonce_j)
K_{f,j} = KDF(K_f || copy_id_{f,j})

mask_{f,j,k} = PRF_to_Zp(K_{f,j},fid || k || ver_{f,k})
m_{f,j,k} = c_{f,k} + mask_{f,j,k} mod p
```

provider 不获得副本 PRF key。保存一个编码副本不能直接计算另一 `copy_id` 下的数据。

### 4.2 版本根与副本分配根

```text
VersionLeaf_{f,k} = H("version" || fid || k || ver_{f,k})
VersionRoot_f = MerkleRoot({VersionLeaf_{f,k}}_{k=1}^n)

provider_ref_q = H(
  "provider-ref" || provider_id_q || EncodePoint(pk_q)
)

ReplicaLeaf_{f,j} = H(
  "replica" || j || copy_id_{f,j} || provider_ref_{provider(f,j)}
)
ReplicaRoot_f = MerkleRoot({ReplicaLeaf_{f,j}}_{j=1}^N)
```

同一 provider 的 `(provider_id_q,pk_q)` 在证明中只出现一次；验证者重算 `provider_ref_q`，完整 `ReplicaVector_f` 只引用固定宽度 reference。可选 registry profile 再用 `ProviderRegistryRoot` 证明 reference 处于冻结注册表，但只有净字节和状态成本优于直接提供 distinct-provider 公钥时才采用。

同一逻辑块的全部副本共享 `ver_{f,k}`。审计策略固定为 `AuditAllReplicas`，验证者必须检查完整 `ReplicaVector_f`，不能由证明方选择子集。

### 4.3 文件状态根

```text
FileMeta_f = (
  owner_id,pk_DO,fid,n,N,
  VersionRoot_f,ReplicaRoot_f
)

FileRoot_f = H("file" || Encode(FileMeta_f))
FileState[fid] = active_FileRoot_f
```

`FileRoot_f` 已表示完整当前文件状态，不再引入每副本认证根或额外 epoch。

## 5. 数据标签

```text
DataCtx_{f,j,k} = (
  owner_id,fid,copy_id_{f,j},k,ver_{f,k}
)

U_{f,j,k} = H_data("data" || Encode(DataCtx_{f,j,k}))

sigma_{f,j,k}
=
(U_{f,j,k} * u^{m_{f,j,k}})^x * D_ID
```

当前版本变化会同时改变副本掩码、数据标签和验证基点，旧块与旧标签不能响应新 active `VersionRoot_f`。

## 6. 前向私有索引与关键词状态认证

### 6.1 前向私有状态链

```text
kw_id = PRF(K_kw,w)
addr_t = H("addr" || kw_id || st_t)
key_t = KDF("index" || kw_id || st_t)
```

每次关系更新选择随机新状态 `st_t`，并在 `addr_t` 下加密：

```text
(fid,op,rel_nonce,st_{t-1})
```

历史 token 只包含旧 head，不能计算未来随机状态对应的地址。删除后重新插入同一文件必须生成新的 `rel_nonce`。

### 6.2 Compressed sparse Merkle 关系集合

当前活动关系为：

```text
RelCtx_{w,f} = (owner_id,kw_id,fid,rel_nonce_{w,f})
rel_key = H("rel-key" || Encode(RelCtx_{w,f}))

RelLeaf(key) = H(
  "csmt-leaf" || key || H("active")
)

RelBranch(prefix_len,prefix,left,right) = H(
  "csmt-branch" || prefix_len || prefix || left || right
)

ResultSetRoot_w = CSMTRoot(
  SortUnique({rel_key -> H("active")})
)
```

CSMT 是由完整 key bit string 构造的确定性二进制 Patricia trie：叶子绑定完整 key；每个内部节点的 `prefix` 是其全部后代 key 从 bit 0 开始的最长公共前缀，`prefix_len` 使用固定宽度整数编码，未使用的末尾 bit 必须为零；左右子树分别包含下一个分歧 bit 为 `0/1` 的非空 key 集合；禁止单子节点、依赖插入顺序的旋转或实现私有节点编号。空集合使用 `H("csmt-empty")`，单叶树直接使用对应 `RelLeaf`。

给定完整 canonical `Res`，验证者按相同规则直接重建 `ResultSetRoot_w`，不接收额外 membership proof。SP 负责以 `O(r log r)` 对 relation key 排序；验证者只做 `O(r)` 相邻严格递增检查，并在线性时间内构建最多 `2r-1` 个 CSMT 节点。单条更新最坏为 `O(lambda)`，平均访问节点数随实际 trie 高度变化。

### 6.3 关键词状态标签

```text
KwCtx_w = (
  owner_id,kw_id,H(st_head),
  ResultSetRoot_w
)

U_kw = H_kw("keyword-state" || Encode(KwCtx_w))
kappa_w = U_kw^x * D_ID
```

关键词 leaf 为：

```text
KeywordLeaf_w = H(
  "keyword" || Encode(KwCtx_w) || EncodePoint(kappa_w)
)

IndexRoot = SparseMerkleRoot(
  {H("keyword-key" || kw_id) -> KeywordLeaf_w}
)
IndexState = active_IndexRoot
```

每次关系更新只重新生成一个 `kappa_w`。不保存逐关系代数标签，也不维护关系标签乘积。

关键词状态首次发布或更新激活时，合约先检查 `kappa_w` 的单标签 pairing 等式。更新还必须提交 `Pi_idx_update`，证明 old/new `IndexRoot` 使用相同 sibling path，且只把目标 `kw_id` 的 old leaf 替换为已认证 new leaf。通过后 relayer 才能激活候选 root，不能改动其他关键词状态。审计阶段仍把同一 `kappa_w` 合入统一数据持有等式，因而不需要额外 owner 交易签名。

## 7. 搜索、冻结快照与挑战

### 7.1 搜索验证

SP 返回：

```text
Res = {(fid_l,rel_nonce_l)}_{l=1}^r
Pi_idx = (KeywordLeaf_w,path_w)
```

冻结前验证者检查：

1. `KeywordLeaf_w -> active_IndexRoot` 路径有效；
2. `Res` canonical 排序，`fid` 和 `RelCtx` 均无重复；
3. 由完整 `Res` 重建的 CSMT root 等于 `ResultSetRoot_w`；
4. 每个 `fid` 存在 active `FileRoot_f`；
5. 每个完整 `ReplicaVector_f` 重建对应 `ReplicaRoot_f`。

### 7.2 冻结查询状态

```text
frozen_IndexRoot = IndexState

StateDigest = H(
  sorted({fid_l || active_FileRoot_{fid_l}}_{l=1}^r)
)

AuditRoot = H(
  "audit" || query_id || owner_id || kw_id ||
  frozen_IndexRoot ||
  StateDigest || c || AuditAllReplicas || beacon_round
)
```

`ExpectedProviderSet` 由全部 frozen `ReplicaVector_f` 中的 provider reference 确定。`StateDigest` 已承诺对应 `FileRoot_f`，每个 `FileRoot_f` 又承诺 `ReplicaRoot_f`，因此无需再在 `AuditRoot` 中增加 `ProviderDigest`。

合约只为未完成查询保存最小共识状态：

```text
QueryState[query_id] = (
  AuditRoot,beacon_round,deadline,phase
)

CommitState[query_id] = (
  commit_bitmap,reveal_bitmap,C_all,batch_round,
  temporary_commitments
)
```

`frozen_IndexRoot`、`StateDigest`、完整文件/副本材料和 `ExpectedProviderSet` 是 `AuditRoot` 的 opening witness，不作为重复共识字段保存。commitment 只在本次查询的 commit/reveal 生命周期内存在，查询完成或中止后清理；bitmap 和 `phase` 防止同一 provider 重复提交或越阶段调用。

`query_id` 全局唯一，完成、失败、随机源超时或协议超时后均进入终态并不得重用。后续文件或索引更新不影响已经冻结的查询；验证者打开 `AuditRoot` 中的 frozen roots，而不是读取验证时的最新 active roots。

### 7.3 挑战

冻结交易位于高度 `h` 时，预先确定 `beacon_round = h + Delta_beacon`。在 `AuditRoot` 冻结后，从该轮 finalized randomness 或可验证 VRF beacon 获得不可预测挑战种子：

```text
seed_chal = H(
  "challenge-seed" || chain_id || contract_address ||
  AuditRoot || beacon_round || FinalizedRandomness(beacon_round)
)

I_f = SampleWithoutReplacement(seed_chal || fid,n_f,c_f)

v_{f,j,k} = H_p(
  "challenge" || seed_chal || fid || copy_id_{f,j} || k
)
```

随机源在 deadline 前不可用或无法验证时，查询只能 `Aborted`，不得更换 `beacon_round` 后沿用原 `query_id`。最新区块哈希可作为弱随机实验配置，但不作为正式安全配置。`seed_chal` 与后续可选的 `seed_batch` 使用不同域标签，不能互换。

同一文件全部副本使用相同块索引，因此每个文件只提交一份版本 multiproof；不同 copy 使用不同挑战系数，不能为降低计算量而复用系数。

## 8. 可提取的标量隐藏响应与 batch

令 `Omega_q` 是 provider `q` 负责的全部 `(f,j,k)`：

```text
S_q = Product_{(f,j,k) in Omega_q}
  sigma_{f,j,k}^{v_{f,j,k}}

mu_q = Sum_{(f,j,k) in Omega_q}
  v_{f,j,k} * m_{f,j,k} mod p

epsilon_q <- Z_p
R_q = u^{epsilon_q}

gamma_q = H_p(
  "mask" || AuditRoot || provider_id_q ||
  S_q || R_q || LocalScope_q
)

z_q = epsilon_q + gamma_q * mu_q mod p
```

`gamma_q` 将随机掩码与固定响应关联。对同一 `S_q,R_q,LocalScope_q` fork 出两个不同 `gamma_q` 后，可由两个接受 transcript 提取 `mu_q`。

provider 提交：

```text
C_q = H(
  "response" || AuditRoot || provider_id_q ||
  S_q || R_q || z_q || LocalScope_q
)
```

只有 `ExpectedProviderSet` 中每个不同 provider 恰好提交一次 commitment 后，合约才冻结：

```text
C_all = H(sorted({provider_id_q || C_q}))

beta_q = H_p(
  "batch" || AuditRoot || C_all || batch_salt || provider_id_q
)
```

默认 Fiat-Shamir profile 取 `batch_round = none`、`batch_salt = H("fs-batch" || AuditRoot || C_all)`，并显式承担下述 `Q_H/(p-1)` grinding 项。强化 profile 在 `C_all` 冻结后固定独立 `batch_round`，由该轮 beacon 导出 `seed_batch`，取 `batch_salt = H("batch-seed" || AuditRoot || C_all || batch_round || seed_batch)`；`seed_batch` 不得复用 `seed_chal`。

provider reveal 并签名：

```text
(query_id,AuditRoot,C_all,batch_salt,LocalScope_q,S_q,R_q,z_q)
```

重复、缺失、集合外 provider，commit/reveal 不一致、scope 重叠或缺口均导致拒绝。失败查询不能用同一 `query_id` 重试。

Fiat-Shamir batch 的 grinding 成功概率在最多 `Q_H` 个候选 commitment digest 下，上界为 `Q_H/(p-1)` 量级。若参与方冻结、查询不可重用或可接受的 grinding bound 不能满足部署要求，必须使用上述 post-commit beacon profile，而不是临时更换 `C_all` 或重试同一查询。

## 9. 统一聚合证明

定义：

```text
Sigma = kappa_w * Product_q S_q^{beta_q * gamma_q}

R = Product_q R_q^{beta_q}

z = Sum_q beta_q * z_q mod p

X_data = Product_{f,j,k in I_f}
  U_{f,j,k}^{beta_{provider(f,j)} * gamma_{provider(f,j)} * v_{f,j,k}}

W = 1 + Sum_{f,j,k in I_f}
  beta_{provider(f,j)} * gamma_{provider(f,j)} * v_{f,j,k}
  mod p
```

代数核心为：

```text
CoreAccumulator = (Sigma,R,z)
```

公开验证等式为：

```text
e(Sigma,g2) * e(R,pk_DO)
=
e(U_kw * X_data * u^z,pk_DO)
*
e(Q_ID^W,mpk)
```

完整证明为：

```text
Pi = (
  query_id,AuditRoot,BeaconOpenings,
  seed_chal,batch_round,batch_salt,
  Res,KeywordLeaf_w,Pi_idx,
  {FileMeta_f,ReplicaVector_f,Pi_ver_f}_{fid in Res},
  {S_q,R_q,z_q,LocalScope_q,C_q,sig_q}_q
)
```

`BeaconOpenings` 包含 challenge `(beacon_round,beacon_value,beacon_proof)`；强化 batch profile 还包含独立 `(batch_round,batch_value,batch_proof)`，默认 Fiat-Shamir profile 不包含第二项。

这里的 `Pi` 是完整自包含公开证明 `Pi_public`。链上验证使用 `Pi_onchain`：凡已由 `QueryState/CommitState` 保存或能在同一调用上下文中重算的字段均不重复提交，但必须在字节统计中分别报告两种口径。event 只作为 off-chain 归档，合约不能读取历史 event，因此不能用 event 替代当前验证所需的 state/calldata。`ReplicaVector_f` 使用固定宽度 canonical 编码；相同 provider 的公钥材料只出现一次，副本条目使用 provider reference。只有在 `ProviderRegistryRoot` 的注册证明总字节数小于重复公钥字节数时才采用 registry profile，并单独报告其状态成本。

验证顺序固定为：

1. 验证 `BeaconOpenings` 中 challenge round 及可选 batch round 的 value/proof，重算 `seed_chal/batch_salt`，并用 proof witness 打开 `QueryState.AuditRoot` 中的 frozen index/file/provider 状态和 `beacon_round`；
2. 验证 canonical 结果和 `Pi_idx`，并由完整 `Res` 重建 `ResultSetRoot_w`；
3. 验证全部 replica assignment 和版本 multiproof；
4. 验证 provider 集合、commitments、签名和 scope partition；
5. 重算 `gamma_q`、`beta_q`、`U_kw`、`X_data`、`W`、`Sigma`、`R` 和 `z`；
6. 检查统一 pairing 等式。

多 owner 查询必须按 owner 分组验证。

## 10. 正确性

数据标签满足：

```text
e(sigma_{f,j,k},g2)
=
e(U_{f,j,k} * u^{m_{f,j,k}},pk_DO)
*
e(Q_ID,mpk)
```

关键词状态标签满足：

```text
e(kappa_w,g2)
=
e(U_kw,pk_DO)
*
e(Q_ID,mpk)
```

对 provider 局部响应：

```text
X_q = Product_{(f,j,k) in Omega_q} U_{f,j,k}^{v_{f,j,k}}
w_q = Sum_{(f,j,k) in Omega_q} v_{f,j,k}

e(S_q^{gamma_q},g2) * e(R_q,pk_DO)
=
e(X_q^{gamma_q} * u^{z_q},pk_DO)
*
e(Q_ID^{gamma_q*w_q},mpk)
```

再按 `beta_q` 聚合，并乘入一个关键词状态标签，即得到第 9 节统一等式。`W` 中的常数 `1` 对应 `kappa_w` 的一个 `D_ID` 因子。

## 11. 动态更新与最小控制状态

候选状态不覆盖 active root：

```text
Active(v) + Pending(v+1) -> Active(v+1)
                         \-> Aborted(v+1)
```

`Pending` 保存候选 root、唯一 `update_id`、目标确认集合和 deadline。“ack 已齐全”由 canonical provider bitmap 重算，不单独保存 `Acked` 状态。

### 11.1 文件块更新

对更新块集合 `U_f`：

1. 递增对应 `ver_{f,k}`；
2. 更新基础密文块；
3. 为全部 `N` 个副本重生成对应编码块和数据标签；
4. 更新 `VersionRoot_f` 并形成候选 `FileRoot_f'`；
5. 全部目标 provider 并行验证；每个 provider 对本 candidate 中负责的全部更新块只签署一个 `UpdateAck`，收齐后原子激活。

```text
密码学更新： O(|U_f|*N)
版本树更新： O(|U_f|*log n)
```

缺少、重复、冲突、过期或重放 ack 均不能激活。初始外包同样需要全部 `StoreAck`。

### 11.2 Provider 重新分配

迁移现有 `copy_id` 时，数据标签不变；新 provider 确认数据后原子激活新 `ReplicaRoot`。生成新 `copy_id` 时必须重生成该副本的全部编码块和数据标签。

### 11.3 关键词关系更新

插入或删除关系时原子更新：

```text
encrypted state-chain record
st_head
ResultSetRoot_w
kappa_w
KeywordLeaf_w
IndexRoot
```

删除必须引用当前 active 精确 `RelCtx`。重复插入、重复删除和旧 nonce 重放均拒绝。`Pi_idx_update` 是激活时验证输入而非持久状态，必须证明只更新目标 keyword leaf。关系集合与索引路径更新均为 `O(lambda)` 哈希，关键词状态标签生成是常数次群运算。

## 12. 状态与复杂度

### 12.1 最小状态分类

| 类型 | 内容 |
|---|---|
| 原始动态数据 | `ver_{f,k}`、replica assignment、`st_w`、active `RelCtx` |
| 长期共识状态 | `FileRoot_f`、`IndexRoot` 和可选 `ProviderRegistryRoot`；其 opening witness 由 off-chain 保存 |
| 查询临时状态 | `AuditRoot`、`beacon_round`、deadline、phase、commit/reveal bitmap 和临时 commitments |
| 更新临时状态 | `update_id`、candidate root、deadline、目标 provider bitmap 和 `Pending/Aborted` |
| 证明材料 | `Res`、`FileMeta`、`ReplicaVector`、Merkle multiproof 和 provider responses |
| 可重算量 | `VersionRoot`、`ReplicaRoot`、`ResultSetRoot`、`StateDigest`、`ExpectedProviderSet`、`C_all` 和挑战系数 |

`VersionRoot`、`ReplicaRoot`、`ResultSetRoot` 和 `kappa_w` 是各自 leaf/opening 中的认证值，不要求合约再为同一对象单独建立顶层 storage slot。`StateDigest`、`ExpectedProviderSet` 和 `C_all` 均可由冻结材料重算，不是独立长期状态。合约实现必须逐项报告 storage word、SSTORE/SLOAD、创建时间和清理时间。

### 12.2 计算复杂度

令 `r` 为结果数、`N` 为平均副本数、`c` 为每文件挑战数、`P` 为 provider 数、`u` 为更新块数。

| 阶段 | 主要成本 |
|---|---|
| 副本与数据标签生成 | `O(nN)` PRF 和 G1 运算 |
| 关系更新 | 最坏 `O(lambda)` CSMT 哈希和 1 个关键词状态标签 |
| 搜索证明生成 | `O(r)` 索引读取和 `O(r log r)` canonical 排序 |
| 搜索集合验证 | `O(r)` 顺序检查和 `O(r)` CSMT 节点哈希 |
| 全部 provider 证明生成 | `O(rNc)` 数据访问和 G1 运算 |
| 关键词状态代数验证 | `O(1)` hash-to-G1 |
| 数据基点验证 | `O(rNc)` hash-to-G1 和 G1 标量运算 |
| 最终等式 | 常数次 multi-pairing |
| `u` 块更新 | `O(uN + u log n)` |

最终 pairing 数为常数，但公开验证总计算仍为 `O(rNc)`。

`O(rNc)` 和 `O(uN)` 是审计全部独立副本的功能代价。实现可按 provider 流式处理并并行执行，使用批量 hash-to-curve 和 Pippenger MSM 降低常数，但不得省略 copy-specific 系数或让未认证的 `X_q` 替代验证者重算。对 `n` 个块中 `b` 个损坏块，不放回抽样的精确检测率和保守挑战数为：

```text
P_detect_exact = 1 - C(n-b,c)/C(n,c)
alpha = b/n
c_conservative = ceil(log(1-P_target) / log(1-alpha))
```

实现从精确超几何式求最小 `c_min`；`c_conservative` 只作为大 `n` 下的便捷保守估计，并同时满足协议设定的上下界。

### 12.3 通信与链上边界

| 材料 | 规模 |
|---|---|
| 搜索结果 | `O(r)` |
| 完整副本分配 | `O(rN)` provider reference；provider 公钥材料按 distinct provider 计 |
| 版本 multiproof | 朴素 `O(rc log n)`，可压缩共享路径 |
| provider 局部响应和签名 | `O(P)` |
| 可重算代数核心 | `2|G1|+|Z_p|` |

部署采用两级核心路径和一个独立扩展：

1. `Direct-Onchain`：只用于实测 gas 上限内的小到中等参数；证明使用 calldata，长期只保留最小状态，详细材料优先进入 event；
2. `Scalable-Onchain`：使用确定性分块累加，保存 chunk index、running accumulator 和完成 bitmap；它降低单交易 gas，不改变总复杂度；
3. `Succinct-Verification Extension`：可用 zkVM/SNARK 证明 CSMT、Merkle 和 MSM 计算，但仅在实际实现并验证后计入结果，不属于当前核心安全声明。

Scalable 与 Direct 路径必须在重叠参数上得到相同结果。实验需对每个 block gas limit 给出最大可执行 `(r,N,c,P)`，并分别报告 calldata、storage、结构哈希、MSM 和 pairing gas。

不得把常数 pairing 表述为常数验证时间，也不得预先声称任意规模可直接链上执行。

## 13. 安全证明义务

正式证明必须分别完成：

1. `IND-CKA Keyword Privacy` 和 `L-Adaptive Forward Privacy`；
2. `CSMT Search-Result Completeness`；
3. `CL-HomAuth-Tag-EUF-CMA-I/II`；
4. `Enhanced Audit-Proof-EUF-CMA` 和 `Local Replica-PoR Knowledge Soundness`；
5. `Active-Version Freshness` 和 `Copy-Specific Replica Independence`；
6. `Commit-Before-Beta FS Batch Soundness`；
7. `Audit-Response Scalar One-Wayness`；
8. `Frozen-Query Consistency`、`Atomic Update Safety` 和 `Provider Accountability`；
9. `Joint Audit Soundness` 与并列隐私性质组成的 Overall Security Profile。

底层假设固定为 asymmetric co-CDH、离散对数、PRF 安全、Merkle 抗碰撞、签名 EUF-CMA、随机预言机和前向私有 SSE 所需的 PRF/KDF/加密安全。
