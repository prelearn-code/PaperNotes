# 核心方案

本文给出 KCVB-Audit 的核心方案主稿。方案定位为：

```text
KCVB-Audit: 关键词搜索触发的多副本联合完整性审计方案
```

方案面向加密去中心化存储网络。用户按关键词搜索密文文件时，系统不只返回搜索结果，还在同一个审计会话、同一个 `AuditRoot` 和同一个挑战上下文下，对搜索结果集合中的所有目标文件副本发起完整性审计。任意公开验证者最终一次性验证：

```text
搜索结果是否完整
+ 返回文件是否完整
+ 每个文件的多个副本是否真实存在
+ 副本是否属于指定 provider
+ 证明是否对应当前链上状态
```

本文的核心不是把所有字段塞入块级标签，而是采用分层最小绑定结构：搜索层证明结果完整性，副本层证明 copy-provider 分配，数据层证明副本块持有性，链上层证明状态新鲜性，`AuditRoot` 将四者统一桥接。

## 1. 问题定位

加密去中心化存储系统中，用户将文件加密后外包给多个独立存储服务提供者。后续用户按关键词搜索密文文件，并希望系统同时保证：

1. 存储节点返回的文件标识集合正好是当前关键词匹配结果；
2. 返回文件的密文块仍被正确保存；
3. 合约约定的多个副本不是由同一份副本冒充；
4. 每个副本属于指定 provider，且 provider 对响应承担责任；
5. 证明绑定最新文件摘要状态、最新搜索状态和当前链上挑战；
6. 审计过程不依赖单一可信 TPA；
7. 链上只保存紧凑摘要、挑战和日志，而不是完整证明或完整索引。

现有工作分别覆盖了部分目标。Song 2025 支持加密去中心化存储中的可验证搜索和完整性审计，但不处理多副本真实性。Zhang 2023 处理去中心化多副本与批量审计，但不证明关键词搜索结果完整性。Zhu 2026 提供证书无关、无 TPA 的轻量审计，但不面向 searchable multi-copy DSN。Li 2022 是经典 identity-based 多副本 PDP 参考，但依赖 TPA，且不支持区块链和搜索。

因此，本文研究问题为：

```text
如何在无可信 TPA 的加密去中心化存储中，将关键词搜索结果集合本身作为审计对象，
在同一个 AuditRoot 下公开验证搜索结果完整性、数据完整性、多副本真实性、
provider 责任绑定和链上状态一致性，并使链上开销保持可部署。
```

本文贡献可以概括为五点。

1. **搜索触发的多副本联合审计模型。** 关键词查询返回结果集合后，系统立即对结果集合中的所有文件副本执行统一审计。
2. **分层最小绑定结构。** `SearchCtx` 绑定搜索完整性，`AssignRoot` 绑定 copy-provider 分配，`FileRoot` 绑定文件与副本摘要状态，`AuditRoot` 桥接搜索与审计。
3. **结果集合级多副本批量证明。** 审计对象从单个文件扩展为 `keyword result set x file copies x providers`。
4. **链上状态一致性与挑战绑定。** 链上挑战绑定 `AuditRoot`，旧搜索结果、旧文件摘要状态和错误 provider 范围不能响应当前挑战。
5. **provider 责任绑定。** 通过 `AssignRoot_f + provider signature` 绑定副本分配与 provider 响应责任。

## 2. 系统模型

系统包含四类核心实体。

| 实体                   | 作用                           | 信任假设                 |
| -------------------- | ---------------------------- | -------------------- |
| KGC                  | 生成系统参数和用户部分私钥                | 诚实但好奇，不能获得用户完整私钥     |
| 数据拥有者 DO             | 加密文件、生成副本、构建索引、生成标签、发起搜索和审计  | 对自身数据可信              |
| 存储服务提供者 SSP/provider | 保存密文副本、标签和加密索引，生成搜索结果与审计响应   | 不可信，可能删除、篡改、遗漏、替换或合谋 |
| 区块链/智能合约 SC          | 记录承诺、检查当前状态、生成挑战、记录审计结果和争议摘要 | 公开、不可篡改、合约代码确定执行     |

任何第三方都可以作为公开验证者读取链上状态和证明并执行 `Verify`。公开验证者不是可信实体，也不是系统必须依赖的固定参与方。如果某个验证者给出错误结论，其他实体可以用同一组公开输入重新验证并驳斥。

第一版方案中，数据拥有者同时作为搜索用户。授权搜索、多用户访问控制、权限撤销和复杂多关键词查询作为后续扩展，不进入核心构造。

## 3. 威胁模型与安全目标

主要攻击者是恶意或理性的 SSP/provider。攻击者可能执行：

1. 删除部分文件块，仅保留标签并试图生成有效证明；
2. 删除部分副本，或多个 provider 共用一份副本冒充多副本；
3. 搜索时遗漏匹配文件；
4. 插入不匹配文件或替换真实搜索结果；
5. 使用旧文件状态、旧索引或旧标签响应当前挑战；
6. 将一个 provider 或一个 copy 的响应迁移到另一个 provider 或 copy；
7. 拒绝对自身响应承担责任；
8. 对外声称错误验证结果。

方案需要满足以下目标。

| 目标 | 含义 |
|---|---|
| 关键词隐私 | SSP 不能从搜索令牌直接恢复明文关键词 |
| 前向安全 | 新插入关键词-文件关系不能被历史搜索令牌直接关联 |
| 搜索结果可验证 | 遗漏、插入、替换结果均会导致索引证明失败 |
| 数据完整性 | 缺失或篡改挑战块无法通过聚合验证 |
| 副本-节点绑定 | 一个副本或节点的证明不能迁移到另一个副本或节点 |
| provider 责任绑定 | 指定 provider 必须对其响应 transcript 签名 |
| 链上状态一致性 | 旧文件状态或旧索引证明不能通过当前链上状态验证 |
| 证书无关不可伪造 | KGC 或外部攻击者均不能单独伪造完整标签 |
| 无 TPA 公开验证 | 审计正确性不依赖单一可信第三方 |

本文不声称隐藏访问模式。默认泄露包括搜索模式、访问模式、结果数量、更新时间、副本数量、provider 分配和链上审计时序。

为避免证书无关审计安全模型中的标签保留型伪造漏洞，本文采用增强型 Proof-EUF-CMA 口径描述数据持有安全。攻击者可以在外包阶段获得合法标签，也可以观察历史挑战和历史证明；但当攻击者声明或模拟某些文件块已经损坏、丢失或不再由 SSP 持有后，不能再对这些块发起新的标签生成查询或证明 oracle 查询。若后续挑战集合包含这些损坏或丢失块，而攻击者仍能在不知道对应数据块线性组合 `mu_{l,j}` 的情况下输出通过 `Verify` 的证明，则攻击者赢得增强型 Proof-EUF-CMA 游戏。

## 4. 核心状态与分层绑定

### 4.1 文件与副本标识

文件 `F` 被划分为 `n` 个块：

```text
F = {m_1, m_2, ..., m_n}
```

数据拥有者为每个文件生成 `N` 个可区分副本：

```text
Copy_j = CopyGen(F, j, rho_j), 1 <= j <= N
```

文件标识定义为：

```text
fid = H(owner_id || file_name || file_nonce)
```

第 `j` 个副本的标识不直接绑定 provider：

```text
copy_id_{f,j} = H("copy" || fid || j || copy_nonce_j)
```

该设计使块级标签只需要识别副本身份和块位置。provider 身份由副本分配根和 provider 签名绑定，而不是重复进入每个块标签。

### 4.2 搜索状态

对每个关键词 `w`，数据拥有者维护最新状态 `st_w^t`。插入新关键词-文件关系时生成新状态，并用加密指针连接历史状态：

```text
ptr_t = Enc(H(st_w^t), st_w^{t-1})
T_w^t = H_kw(w || st_w^t)
```

为了公开验证搜索结果完整性，数据拥有者为每个关键词令牌 `kw_tag` 维护结果集合承诺：

```text
RRoot_w = MerkleRoot(sorted({fid || status}_{fid in ResultSet_w}))
IndexRoot = MerkleRoot({kw_tag || RRoot_w})
SearchCtx = H("search" || IndexRoot || epoch)
```

其中 `status` 表示有效、修改后有效或 tombstone 删除。验证者根据返回集合和 tombstone 集合重算：

```text
RRoot'_w = MerkleRoot(sorted({fid || status}_{fid in Res union Tomb_w}))
```

并检查 `RRoot'_w = RRoot_w`，再验证 `kw_tag -> RRoot_w -> IndexRoot -> SearchCtx`。因此 `Pi_idx` 证明完整结果集合，而不是只证明返回元素属于集合。

搜索状态只证明 `Res` 是关键词 `w` 当前对应的完整结果集合，不包含 `copy_id`、`provider_id`、标签或挑战信息。

### 4.3 副本与 provider 分配

每个文件维护副本分配根：

```text
AssignRoot_f = MerkleRoot(sorted({
  copy_id_{f,j} || provider_id_j || pk_provider_j
}_{j=1}^N))
```

`AssignRoot_f` 的作用是证明某个 copy 确实被分配给某个 provider。若攻击者将一个 provider 的响应迁移给另一个 provider，则 `copy_id || provider_id || pk_provider` 的成员证明或 provider 签名会失败。

### 4.4 文件状态根

每个副本维护标签根：

```text
TagRoot_{f,j} = MerkleRoot({H(sigma_{f,j,k})}_{k=1}^n)
```

所有副本形成副本集合根：

```text
CopySetRoot_f = MerkleRoot(sorted({
  copy_id_{f,j} || TagRoot_{f,j}
}_{j=1}^N))
```

文件状态根为：

```text
FileRoot_f = H(
  "file" ||
  fid ||
  AssignRoot_f ||
  CopySetRoot_f
)
```

`FileRoot_f` 绑定文件身份、副本集合、provider 分配和标签集合。当前状态一致性由 `FileRoot_f` 和链上 `FileStateTable[fid]` 保证，不需要额外记录文件版本号。

### 4.5 联合审计根

搜索得到结果集合：

```text
Res = {fid_1, fid_2, ..., fid_r}
```

构造结果文件根集合：

```text
ResultFileRoot = MerkleRoot(sorted({
  fid_l || FileRoot_{fid_l}
}_{fid_l in Res}))
```

联合审计根定义为：

```text
AuditRoot = H(
  "audit" ||
  query_id ||
  SearchCtx ||
  RRoot_w ||
  ResultFileRoot ||
  ProviderScope ||
  audit_policy
)
```

`AuditRoot` 是本文核心桥接结构，同时绑定关键词搜索状态、搜索结果集合、结果文件的当前摘要状态、每个文件的副本状态、本次要审计的 provider 范围和审计策略。挑战必须绑定 `AuditRoot`，否则 SSP 可能先观察挑战再选择性返回搜索结果。

链上保存：

```text
FileStateTable[fid] = FileRoot_f
SearchState[owner_id] = SearchCtx
AssignmentDigest[fid] = AssignRoot_f
ChallengeTable[query_id] = chal
AuditLog[query_id] = audit_result
```

## 5. 证书无关同态认证标签

令 `G1`、`G2`、`GT` 为素数阶 `p` 的双线性群，`e: G1 x G2 -> GT`。`g1 in G1`、`g2 in G2` 为生成元。KGC 选择主密钥 `s`：

```text
msk = s
mpk = g2^s
Q_ID = H_id(ID) in G1
D_ID = Q_ID^s
```

数据拥有者选择秘密值 `x_DO`：

```text
x_DO <- Z_p
pk_DO = g2^{x_DO}
sk_DO = (D_ID, x_DO)
```

对文件 `fid` 的第 `j` 个副本中第 `k` 个块，定义最小块绑定上下文：

```text
B_{f,j,k} = H1("block" || copy_id_{f,j} || k)
```

认证标签为：

```text
sigma_{f,j,k}
=
(H2(B_{f,j,k}) * u^{m_{f,j,k}})^{x_DO} * D_ID
```

其中 `u in G1` 是公共随机群元素。块级标签仅绑定 `copy_id`、块位置和数据块，用于证明指定副本块的持有性。文件摘要状态、副本分配、provider 身份和搜索状态分别由 `FileRoot_f`、`AssignRoot_f`、provider 签名和 `AuditRoot` 绑定。

## 6. 方案算法

KCVB-Audit 包含以下算法：

```text
Setup(lambda) -> (PP, msk, mpk, SC)
PartialKeyGen(msk, ID) -> D_ID
UserKeyGen(D_ID) -> (sk_DO, pk_DO)
ProviderKeyGen(provider_id) -> (sk_provider, pk_provider)
Outsource(F, KW, N, ProviderSet) -> receipt
TokenGen(sk_DO, w) -> (T_w, st_w)
Search(T_w, st_w, EDB) -> (Res, Pi_idx, RRoot_w)
BuildAuditContext(Res, RRoot_w, ProviderScope) -> AuditRoot
ChallengeGen(query_id, AuditRoot, c) -> chal
ProofGen(Res, Pi_idx, AuditRoot, chal, Copies, Tags) -> Pi
Verify(Pi, chal, pk_DO, public_state) -> {0,1}
Update(op, fid, delta, KW_delta) -> update_receipt
```

### 6.1 Setup

KGC 初始化双线性群参数、哈希函数、主密钥和主公钥。区块链部署审计合约。合约只记录公共参数摘要、用户公钥、provider 公钥、状态根、挑战、证明摘要和审计日志。

### 6.2 KeyGen

KGC 运行 `PartialKeyGen` 生成 `D_ID = H_id(ID)^s`。数据拥有者选择 `x_DO`，得到 `pk_DO = g2^{x_DO}` 和 `sk_DO = (D_ID, x_DO)`。KGC 不知道 `x_DO`，因此不存在 identity-based key escrow。

每个 provider 生成签名密钥：

```text
(sk_provider_j, pk_provider_j) <- ProviderKeyGen(provider_id_j)
```

`pk_provider_j` 登记到链上或由链上 provider 注册状态承诺。

### 6.3 Outsource

数据拥有者执行：

1. 加密文件并切块；
2. 生成 `N` 个可区分副本；
3. 为每个文件副本生成 `copy_id_{f,j}`；
4. 为每个副本块生成最小同态标签；
5. 构建 `TagRoot_{f,j}`、`CopySetRoot_f`、`AssignRoot_f`、`FileRoot_f`；
6. 构建关键词状态链、`RRoot_w`、`IndexRoot` 和 `SearchCtx`；
7. 将密文副本、标签和索引片段发送给 SSP/provider；
8. 将 `FileRoot_f`、`SearchCtx`、`AssignRoot_f` 和相关摘要写入链上。

链上不保存完整文件、完整索引或完整标签。

### 6.4 TokenGen 与 Search

数据拥有者为关键词 `w` 生成：

```text
T_w = H_kw(w || st_w)
```

并将 `(T_w, st_w)` 发送给 SSP。SSP 沿状态链检索匹配文件，得到：

```text
Res = {fid_1, fid_2, ..., fid_r}
```

同时返回：

```text
Pi_idx = (kw_tag, kw_path, RRoot_w, Res, StatusList, Tomb_w, index_digest)
```

`Pi_idx` 用于证明 `Res` 是当前 `IndexRoot/SearchCtx` 下的完整有效结果集合。

### 6.5 BuildAuditContext

对 `Res` 中每个文件读取链上最新状态：

```text
FileStateTable[fid_l] = FileRoot_{fid_l}
```

验证者或合约根据当前文件状态构造：

```text
ResultFileRoot = MerkleRoot(sorted({
  fid_l || FileRoot_{fid_l}
}_{fid_l in Res}))
```

并生成：

```text
AuditRoot = H(
  "audit" ||
  query_id ||
  SearchCtx ||
  RRoot_w ||
  ResultFileRoot ||
  ProviderScope ||
  audit_policy
)
```

智能合约在登记审计请求时检查：

1. `SearchCtx == SearchState[owner_id]`；
2. `RRoot_w` 能在当前 `IndexRoot/SearchCtx` 下被打开；
3. `FileRoot_{fid_l} == FileStateTable[fid_l]`；
4. `ProviderScope` 中每个 `(fid_l, copy_id_{l,j}, provider_id_j, pk_provider_j)` 均属于对应文件当前 `AssignRoot_f`；
5. `AuditRoot` 能由上述状态重新计算。

任一检查失败时，合约拒绝进入挑战阶段。

### 6.6 ChallengeGen

挑战随机性由合约随机源接口提供，而不是直接依赖 `block.timestamp`、`block.difficulty` 或单一 SSP 提供的 nonce：

```text
rand = RandomBeacon(query_id)
seed = H("chal" || rand || AuditRoot || block_no)
I = {i_1, i_2, ..., i_c}
{v_k}_{k in I}
chal = (query_id, AuditRoot, seed, I, {v_k}, ProviderScope, block_no)
```

`RandomBeacon` 可以按部署场景实例化。Ethereum-like 公链默认使用 EIP-4399 `PREVRANDAO` 作为共识随机源；高价值审计可以使用 Chainlink VRF 或等价可验证随机函数；联盟链或本地实验环境可以采用 commit-reveal 加延迟挑战机制模拟随机源接口。本文不把本地链伪随机数等同于公链安全随机性。

挑战绑定 `AuditRoot`，因此旧搜索状态、旧文件状态、旧副本分配或错误 provider 范围不能响应当前挑战。

### 6.7 ProofGen

对每个 `fid_l in Res`、每个 `copy_j in ProviderScope`，对应 provider 定位挑战块和标签：

```text
m_{l,j,k}, sigma_{l,j,k}, copy_id_{l,j}, provider_id_j
```

并计算局部同态响应：

```text
mu_{l,j} = Sum_{k in I} v_k * m_{l,j,k} mod p
Sigma_{l,j} = Product_{k in I} sigma_{l,j,k}^{v_k}
```

批处理系数为：

```text
beta_{l,j} = H_beta(seed || fid_l || copy_id_{l,j} || provider_id_j)
```

每个 provider 先生成 provider 级聚合响应：

```text
Sigma_provider_j = Product_{fid_l in Res_j} Sigma_{l,j}^{beta_{l,j}}
mu_provider_j = Sum_{fid_l in Res_j} beta_{l,j} * mu_{l,j} mod p
transcript_j = H(AuditRoot || chal || provider_id_j || Sigma_provider_j || mu_provider_j || LocalScope_j)
sig_provider_j = Sign(sk_provider_j, transcript_j)
```

其中 `LocalScope_j` 是该 provider 在本次 `ProviderScope` 中负责的 `(fid_l, copy_id_{l,j})` 集合。provider 签名表示指定 provider 对本次响应负责。

全局聚合响应为：

```text
Sigma_batch = Product_{provider_j in ProviderScope} Sigma_provider_j
mu_batch = Sum_{provider_j in ProviderScope} mu_provider_j mod p
batch_domain = H("KCVB-Batch" || query_id || AuditRoot || ProviderScope || owner_id)
```

统一证明为：

```text
Pi = (
  Res,
  Pi_idx,
  RRoot_w,
  AuditRoot,
  Sigma_batch,
  mu_batch,
  batch_domain,
  {
    provider_id_j,
    pk_provider_j,
    Sigma_provider_j,
    mu_provider_j,
    LocalScope_j,
    sig_provider_j
  }_{provider_j in ProviderScope},
  {
    fid_l,
    FileRoot_{fid_l},
    copy_id_{l,j},
    TagRoot_{l,j},
    Pi_assign_{l,j},
    Pi_file_l,
    Pi_tag_{l,j}
  }_{fid_l in Res, copy_j in ProviderScope},
  aux_digest
)
```

`Pi_assign` 证明 `copy_id || provider_id || pk_provider` 属于 `AssignRoot_f`，`Pi_file` 证明 `FileRoot_f` 是链上最新文件状态根，`Pi_tag` 证明 `copy_id || TagRoot` 属于 `CopySetRoot_f`。若批处理验证失败且需要定位具体失败文件或副本，验证者可以要求 provider 提交更细粒度分组证明，或按文件/副本执行二分定位。

### 6.8 Verify

验证者首先检查链上状态：

1. `chal` 是否存在于 `ChallengeTable[query_id]`；
2. `AuditRoot` 是否与当前 `SearchCtx`、`RRoot_w`、`ResultFileRoot`、`ProviderScope` 和 `audit_policy` 一致；
3. 每个 `FileRoot_f` 是否为 `FileStateTable[fid]` 中的当前状态；
4. `pk_DO` 是否匹配文件登记拥有者；
5. `ProviderScope` 是否匹配当前副本分配状态。

随后验证 `Pi_idx`：

```text
RRoot'_w = MerkleRoot(sorted({fid || status}_{fid in Res union Tomb_w}))
```

检查 `RRoot'_w = RRoot_w`，并检查 `kw_tag -> RRoot_w -> IndexRoot -> SearchCtx`。

然后检查每个副本和 provider：

1. `copy_id_{l,j} || provider_id_j || pk_provider_j` 属于 `AssignRoot_{fid_l}`；
2. `copy_id_{l,j} || TagRoot_{l,j}` 属于 `CopySetRoot_{fid_l}`；
3. `"file" || fid_l || AssignRoot_{fid_l} || CopySetRoot_{fid_l}` 能重建 `FileRoot_{fid_l}`；
4. `transcript_j = H(AuditRoot || chal || provider_id_j || Sigma_provider_j || mu_provider_j || LocalScope_j)`；
5. `VerifySig(pk_provider_j, transcript_j, sig_provider_j) = 1`；
6. `Sigma_batch = Product_j Sigma_provider_j` 且 `mu_batch = Sum_j mu_provider_j mod p`。

最后对每个 `(fid_l, copy_id_{l,j}, provider_id_j)` 重新计算批处理系数和块绑定上下文：

```text
beta_{l,j} = H_beta(seed || fid_l || copy_id_{l,j} || provider_id_j)
B_{l,j,k} = H1("block" || copy_id_{l,j} || k)
```

并检查全局批处理 pairing 等式：

```text
e(Sigma_batch, g2)
=
e(
  Product_{fid_l in Res, copy_j in ProviderScope, k in I}
    H2(B_{l,j,k})^{v_k * beta_{l,j}}
  * u^{mu_batch},
  pk_DO
)
*
e(
  Q_ID^{Sum_{fid_l in Res, copy_j in ProviderScope} beta_{l,j} * Sum_{k in I} v_k},
  mpk
)
```

该批处理验证只在同一个 `ID/pk_DO/mpk` 下执行。若一次查询包含多个数据拥有者，验证者必须按 owner 分组分别构造 `Sigma_batch` 和 `mu_batch`。`beta_{l,j}` 由链上挑战 seed 派生，SSP/provider 不能在挑战前选择批处理系数。批处理验证失败时，验证者只能确定至少一个文件或副本无效；若需要责任定位，可退化到分组验证或二分定位。

若全部检查通过，输出 `1`；否则输出 `0`。失败类型可分为：

```text
FailType in {IndexFail, DataFail, CopyFail, ProviderFail, StateFail, ChallengeFail}
```

## 7. 动态更新

第一版构造不单独记录文件版本号，而是用当前 `FileRoot_f` 表示文件摘要状态。因此，动态更新遵循以下口径。

### 7.1 纯索引更新

如果只新增或删除关键词-文件关系，而文件密文块不变，则只更新关键词状态、`IndexRoot` 和 `SearchCtx`。未修改文件块的标签、`TagRoot_{f,j}`、`CopySetRoot_f` 和 `FileRoot_f` 不需要重建。

### 7.2 新文件插入

插入新文件时，为新文件生成新副本、新标签、新 `TagRoot_{f,j}`、新 `CopySetRoot_f`、新 `AssignRoot_f`、新 `FileRoot_f` 和新的索引条目。新关键词状态保证历史搜索令牌不能直接关联新插入文件。

### 7.3 文件内容修改

如果文件内容发生修改，第一版可证明构造需要重新生成该文件受影响副本的相关标签，并更新：

```text
TagRoot_{f,j}'
CopySetRoot_f'
AssignRoot_f'
FileRoot_f'
SearchCtx'
```

这会带来较高更新成本，但安全语义清晰：旧块、旧标签或旧证明不能通过当前 `FileRoot_f'` 和 `AuditRoot'`。

若后续需要将修改成本优化为只与受影响块数量相关，需要引入局部块上下文：

```text
BlockCtx_{f,k} = H("blockctx" || fid || k || copy_id || update_nonce_k)
```

该优化不作为第一版核心构造。

### 7.4 删除与 tombstone

第一版采用 tombstone 逻辑删除：

```text
del_marker = H(fid || "deleted" || delete_nonce)
```

删除后，索引状态标记为 tombstone，`IndexRoot` 和 `SearchCtx` 更新。逻辑删除不要求立即删除密文块，也不要求重建未修改数据块标签。tombstone 用于区分“合法删除”与“SSP 恶意遗漏结果”。

长期动态场景中，tombstone 会增加索引大小和 `Pi_idx` 完整打开成本。可设置：

```text
tombstone_ratio = |Tomb_w| / (|Res_w| + |Tomb_w|)
```

当 `tombstone_ratio > tau` 时触发索引压缩。压缩只重建关键词结果集合承诺、`IndexRoot` 和 `SearchCtx`；若不物理清理密文块，则通常不重建 `TagRoot_{f,j}`、`CopySetRoot_f` 或 `FileRoot_f`。

## 8. 智能合约接口

智能合约提供：

```text
RegisterUser(ID, pk_DO) -> uid
RegisterProvider(provider_id, pk_provider) -> provider_receipt
PublishFileState(fid, FileRoot_f, AssignRoot_f, CopySetRoot_f) -> receipt
PublishSearchState(owner_id, SearchCtx, IndexRoot) -> receipt
RequestAudit(query_digest, ResultFileRoot, SearchCtx, RRoot_w, ProviderScope, c) -> query_id
RandomBeacon(query_id) -> rand
GenerateChallenge(query_id, AuditRoot) -> chal
SubmitAuditResult(query_id, proof_digest, result, fail_type) -> audit_receipt
ChallengeAuditResult(query_id, dispute_digest) -> dispute_id
UpdateFileState(fid, new_FileRoot_f, update_digest) -> update_receipt
UpdateSearchState(owner_id, new_SearchCtx, update_digest) -> update_receipt
```

`RequestAudit` 只登记查询摘要、结果文件根集合摘要、搜索状态摘要、关键词结果根和审计范围。`GenerateChallenge` 是合约内部的强制检查点：只有当当前 `SearchCtx`、`RRoot_w`、`ResultFileRoot`、`ProviderScope` 和每个文件的 `AssignRoot_f` 均与链上状态一致时，合约才接受 `AuditRoot` 并生成随机挑战写入 `ChallengeTable`。因此，挑战生成本身承担搜索结果约束、副本范围约束和当前状态一致性约束，而不是只提供随机数。

`RandomBeacon` 抽象合约可用的公开随机源。公链实现可读取 EIP-4399 `PREVRANDAO`，高价值审计可接入 Chainlink VRF 或等价 VRF，联盟链和本地 Ganache 实验只把 commit-reveal 加延迟挑战作为工程模拟。无论采用哪种实例化，`block.timestamp`、旧式 `block.difficulty` 或 SSP 单方 nonce 都不能作为唯一随机源。

合约采用状态裁剪机制，只维护当前可用于挑战生成的活跃状态。形式化地，对每个文件和数据拥有者，合约活跃状态为：

```text
ActiveState_SC(fid, owner_id) = {
  FileStateTable[fid] = FileRoot_f,
  SearchState[owner_id] = SearchCtx,
  AssignmentDigest[fid] = AssignRoot_f
}
```

其中 `FileStateTable[fid]` 只保存当前 `FileRoot_f`，`SearchState[owner_id]` 只保存最新 `SearchCtx`，`AssignmentDigest[fid]` 只保存当前副本分配根。`UpdateFileState` 和 `UpdateSearchState` 采用覆盖式更新，旧文件根不再作为当前挑战的可接受输入。历史状态通过事件日志、`update_digest`、`audit_digest` 和争议摘要保留可追溯证据，但不占用合约的活跃验证状态。

因此，合约的 active contract storage 对每个文件和每个数据拥有者保持 `O(1)`。历史日志会随审计和更新次数增长，但它们只用于事后追溯，不参与当前 `GenerateChallenge` 的状态匹配。本文讨论链上存储开销时区分 active state storage 与 append-only event log。

默认模式是链下重密码验证和链上摘要/结果记录。合约负责挑战生成、状态一致性检查、摘要记录和争议入口，不默认执行完整 pairing 验证。

## 9. 正确性与安全直觉

如果 SSP 遗漏匹配文件，验证者由 `Res union Tomb_w` 重算的 `RRoot'_w` 与 `IndexRoot` 下承诺的 `RRoot_w` 不一致。

如果 SSP 插入或替换搜索结果，`Pi_idx` 无法通过，或者对应文件的 `FileRoot_f` 与当前 `FileStateTable[fid]` 不一致，导致 `AuditRoot` 无法重建。

如果 SSP 删除数据但保留标签，它仍然缺少挑战块线性组合 `mu_{l,j}`，无法满足 pairing 聚合等式。

如果 SSP 用一份副本冒充多份副本，则 `copy_id` 与 `CopySetRoot_f` 或 `AssignRoot_f` 不匹配，重新计算的 `B_{l,j,k}` 与标签生成时不同，验证失败。

如果 SSP 将一个 provider 的响应迁移到另一个 provider，则 `copy_id || provider_id || pk_provider` 的成员证明或 provider 签名验证失败。

如果 SSP 重放旧证明，当前挑战绑定 `AuditRoot`，而 `AuditRoot` 进一步绑定当前 `SearchCtx`、`RRoot_w`、`ResultFileRoot` 和 `ProviderScope`。旧索引、旧文件根或旧副本分配均无法通过当前验证。

在状态裁剪机制下，旧 `FileRoot_f`、旧 `SearchCtx` 和旧 `AssignRoot_f` 即使仍可在历史事件日志中追溯，也不能被 `GenerateChallenge` 接受为当前活跃状态。旧证明只能用于复查历史审计记录，不能响应当前挑战。

如果公开验证者谎称错误结果，该声明不会改变证明或链上状态。任何其他实体都可以使用公开输入重复验证。

## 10. 安全定义概要

**数据持有可靠性。** 攻击者若在不知道挑战块线性组合的情况下输出可接受的 `(Sigma_batch, mu_batch)`，则等价于伪造同态认证标签、猜中挑战后的随机线性批处理关系，或求解底层 pairing 困难问题。

**增强型 Proof-EUF-CMA / 数据持有不可伪造性。** 攻击者可以获得外包阶段产生的合法标签和历史证明，但不能在声明某些块已经损坏或丢失后继续对这些块查询新的标签或证明 oracle。挑战由合约在当前 `AuditRoot` 下随机生成，且挑战集合可能包含损坏或丢失块。若攻击者在不知道这些块对应局部线性组合 `mu_{l,j} = Sum_{k in I} v_k * m_{l,j,k}` 的情况下仍输出被 `Verify` 接受的批处理响应 `(Sigma_batch, mu_batch)`，则攻击者获胜。本文标签公式把 `u^{m_{l,j,k}}` 纳入同态认证，验证等式又显式检查 `u^{mu_batch}`，因此仅保留标签和上下文哈希不能替代真实数据块。

**随机线性批处理可靠性。** 批处理系数 `beta_{l,j}` 由链上 `seed`、文件标识、副本标识和 provider 标识共同派生，并且在 SSP/provider 提交证明前不可预测。若至少一个局部响应无效，则其错误项被随机 `beta_{l,j}` 线性抵消并仍通过全局 pairing 等式的概率至多为 `1/p` 量级。该定义只保证批处理接受/拒绝的可靠性；失败定位需要额外的分组验证或二分定位流程。

**搜索结果完整性。** 攻击者若遗漏、插入或替换结果而仍使 `Pi_idx` 通过，则需要找到 Merkle root 碰撞或伪造 `kw_tag -> RRoot_w -> IndexRoot` 的认证路径。

**副本-节点绑定。** 攻击者若将一个副本或节点的证明迁移到另一个副本或节点，则 `copy_id/provider_id/pk_provider/AssignRoot_f` 不一致，或 provider 签名验证失败。

**链上状态一致性。** 攻击者若用旧证明响应新挑战，则必须使旧 `FileRoot_f` 或旧 `SearchCtx` 与当前链上状态一致，这与链上状态不可篡改和哈希抗碰撞矛盾。

**证书无关不可伪造性。** Type-I 攻击者不知道 KGC 主密钥，Type-II 攻击者知道 KGC 主密钥但不知道用户秘密值 `x_DO`。两类攻击者均无法生成同时满足 `mpk` 分量和 `pk_DO` 分量的新标签。

## 11. 实验与对比口径

实验不应声称本文方案在所有单项性能上优于现有方案。本文优势在于搜索触发的多副本联合审计能力和分层统一绑定语义，额外开销需要被明确量化。

本文 concrete construction 采用 pairing-based 同态认证标签。与 Zhu 2026 的 pairing-free 轻量证书无关审计相比，本文预期在 `TagGen` 和 `Verify` 上具有更高链下计算成本。该成本来自 `H2(B)` 到群映射、`G1` 指数运算、`Sigma` 聚合以及 pairing 验证。实验叙述必须明确这一 trade-off：本文不以轻量性优于 Zhu 2026，而是用额外 pairing 成本换取搜索结果集合、多副本分配、provider 责任、链上状态一致性和证书无关标签的统一审计。

建议 baseline 分工如下：

| Baseline | 作用 |
|---|---|
| Song 2025 | 对比可验证搜索和搜索审计开销 |
| Zhang 2023 | 对比多副本、批量审计和链上开销 |
| Zhu 2026 | 对比证书无关、无 TPA 和轻量审计 |
| Li 2022 | 对比经典 identity-based 多副本 PDP |
| Dynamic auditing baseline | 只对比动态更新相关指标 |

功能实验首先给出联合安全矩阵。该矩阵不是将 baseline 描述为弱方案，而是说明它们原本不覆盖完整 searchable multi-copy DSN lifecycle。建议矩阵至少包含以下维度：

| Capability | Song 2025 | Zhang 2023 | Zhu 2026 | Li 2022 | Proposed |
|---|---|---|---|---|---|
| Verifiable encrypted search | Yes | No | No | No | Yes |
| Search result completeness | Yes | No | No | No | Yes |
| Search-triggered multi-copy audit | No | No | No | No | Yes |
| Multi-copy auditing | No | Yes | No | Yes | Yes |
| Copy-provider binding | No | Yes | No | Partial | Yes |
| Provider accountability | No | Limited | No | No | Yes |
| Current state binding | Yes | Limited | Limited | No | Yes |
| Certificateless | No | No | Yes | No | Yes |
| TPA-free public verification | Yes | Yes | Yes | No | Yes |
| Unified AuditRoot binding | No | No | No | No | Yes |

该表用于限定公平比较边界：Song 2025 只作为 searchable auditing baseline，Zhang 2023 只作为 multi-copy/batch baseline，Zhu 2026 只作为 certificateless TPA-free lightweight baseline，Li 2022 只作为 classical multi-copy PDP baseline。本文不在 baseline 不支持的能力上宣称性能优越。

必要实验包括：

1. 功能与安全属性表；
2. 理论复杂度；
3. 链下计算开销；
4. 通信开销和 proof size；
5. 存储开销；
6. 链上 gas 和 transaction payload；
7. 动态更新；
8. 分组验证和可扩展性；
9. 检测概率参数分析。

不设置额外攻击检测实验。攻击场景通过安全定义、证明概要和正确性测试说明。消融实验不作为必要主实验；若需要解释开销来源，可拆分报告 `Pi_idx`、`Pi_assign`、`Pi_file`、`Pi_tag`、provider 签名、`Sigma/mu` 和 pairing 验证时间。

链下计算实验需要单独报告 `hash-to-G1/H2(B)` 时间、`G1` 指数运算时间、局部 `Sigma_{l,j}`/`mu_{l,j}` 生成时间、provider 级聚合时间、`Sigma_batch`/`mu_batch` 批处理聚合时间、provider 签名生成与验证时间、pairing 次数、pairing 总时间、Merkle/索引证明验证时间和链上状态检查时间。主验证口径应报告每个 owner/group 的 pairing 次数是否保持常数，以及 `r x N` 增长时的摊薄验证时间。

通信实验应增加多业务并发总带宽口径。除分别报告单项 proof size 外，还应构造 `Composed Workflow` 对照：用 Song-style search proof 加 Zhang/Li-style multi-copy audit proof，再加当前文件摘要、provider 签名和链上摘要元数据，模拟现有模块化方案同时完成搜索检索、多副本完整性审计、provider 责任绑定和链上状态一致性验证所需传输的总证明材料。本文方案与该组合工作流比较 `total proof material`、`total transmitted bytes` 和链上 payload。

链上成本实验应单独报告 active state update gas。重点测量 `active_state_slots`、`update_file_state_gas`、`update_search_state_gas`、`generate_challenge_gas`、`submit_audit_result_gas` 和 `event_log_bytes`。由于链上只替换固定数量的 roots/digests，并记录摘要型日志，当前活跃状态更新 gas 不应随文件块数 `n`、搜索结果数 `r` 或副本数 `N` 线性增长。需要同时说明：链下重标签、索引重建、`Pi_idx` 打开、`Pi_assign/Pi_file/Pi_tag` 集合、provider 签名和完整 proof size 仍会随规模增长；本文的 `O(1)` 仅指 active on-chain state storage 和固定摘要更新，不指全部系统开销为常数。

## 12. 第一版限定与后续扩展

第一版限定：

- 单关键词搜索；
- 数据拥有者自己发起查询；
- tombstone 逻辑删除；
- 不单独维护文件版本号；
- 文件内容修改触发该文件相关标签和 `FileRoot_f` 更新；
- 链下重密码验证，链上记录摘要和结果；
- 支持分组审计和分组验证，但不声称 constant-size batch proof。

后续扩展：

- 局部块上下文，用于局部重标签；
- 多关键词查询；
- 授权搜索和权限撤销；
- 访问模式隐藏和结果填充；
- 更紧凑的多文件多副本聚合证明；
- 可在链上高效验证的简洁证明；
- 将多项式承诺融合到 `ResultFileRoot/CopySetRoot`，压缩多文件多副本证明大小。
